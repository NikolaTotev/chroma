# Chroma — Decisions Log

> The project's memory (ORCHESTRATION.md §7). One line per non-obvious
> cross-crate decision: date, decision, why. Append-only.

| Date | Decision | Why |
|------|----------|-----|
| 2026-06-22 | **Dev environment: Windows-first, capture deferred.** Scaffold and build the platform-agnostic crates on Windows (`x86_64-pc-windows-msvc`); the Linux-only `chroma-capture-x11` / `chroma-capture-wayland` backends are excluded from the workspace members until built on Linux/WSL. | Spec targets Ubuntu, but the contracts (M0) and most logic crates are platform-agnostic and can progress now. Capture is M1/M7 work. WSL2 (Ubuntu-24.04) is available on this machine for the capture milestones. |
| 2026-06-22 | **Contracts authored as v0.1 DRAFT, not yet frozen.** The three `-api` crates compile with fakes and clean `cargo doc`, but are not declared frozen until the M0 gate is formally closed. | ORCHESTRATION.md §10: Architect drafts → Orchestrator freezes at the M0 gate. Until frozen, contract shapes may still be revised without a CONTRACT-CHANGE event. |
| 2026-06-22 | **`Project` stores serializable `ModifierSpec` data, not `dyn Modifier` trait objects.** `chroma-modifiers` will construct `Box<dyn Modifier>` from a `ModifierSpec` at load time. | Reconciles two spec invariants: the project model must be a plain serializable value type (§3.2), while `Modifier` is a behavior trait (§4.3). Keeping data and behavior separate lets `chroma-project` (de)serialize without depending on any modifier implementation. |
| 2026-06-22 | **`CompositePass` is a trait in `chroma-core-api`** (a drawing sink), passed to `Modifier::paint` as `&mut dyn CompositePass`. | Overlay modifiers must paint without `chroma-core-api` depending on `chroma-compositor`. The compositor implements the trait; modifiers depend only on the contract. Preserves the "consumers depend on `-api` only" rule (§0). |
| 2026-06-22 | **`TimeStamp` is `u64` nanoseconds on a single monotonic clock.** | CAP-05 / ORCHESTRATION "One clock" invariant: capture frames and input events share one monotonic timebase. Nanoseconds give ±1-frame resolution headroom at any practical fps. |
| 2026-06-22 | **Errors are hand-rolled per-crate enums (no `thiserror`/`anyhow`).** | Keeps the contract crates' dependency surface minimal (engineering-style: minimal deps); the enums are small and the boilerplate is contained. |
| 2026-06-22 | **`chroma-capture-x11` (M1) added as a workspace member with target-gated deps.** Real impl under `#[cfg(target_os = "linux")]`; a stub elsewhere returns `Unavailable`. `x11rb` is the only backend dependency. | Lets the single workspace build on Windows (stub) and Linux/WSL (real) without a separate manifest. `x11rb`'s pure-Rust connection needs no libxcb/system-X headers, honoring minimal-deps. |
| 2026-06-22 | **X11 frame capture uses core `GetImage` (ZPixmap), not MIT-SHM, for M1.** SHM (zero-copy) is noted as a later internal optimization behind the same `ScreenCapturer` contract. | Gets a correct, dependency-light capture path working first; SHM adds SysV shared-memory + `unsafe` and can be swapped in without any consumer change (spec CAP-06 / §3.1). |
| 2026-06-22 | **Single timebase via a process-global monotonic origin** (`std::time::Instant` in a `OnceLock`), shared by capturer and event source; events are stamped at receive time, frames at grab time. | Satisfies CAP-05 "one clock" without `libc` and without the two subsystems sharing an instance. Stamping events at receive (vs. correlating X server time) is an accepted M1 simplification. |
| 2026-06-22 | **Dev/test environment is WSLg = rootless Xwayland; desktop root `GetImage` returns `BadMatch`.** Pixel-path tests capture a server-backed pixmap we create; root capture is a best-effort/skip test. | Rootless Xwayland has no composited root pixmap to read — the same Wayland limitation the spec addresses with PipeWire (M7). The X11 backend targets native X11 sessions, where root capture works; tests stay deterministic in either env. |
| 2026-06-22 | **M2 compositor is a CPU reference implementation first, wgpu deferred.** `chroma-compositor::CpuCompositor` implements `Compositor` in software; a wgpu compositor is a later swap behind the same trait. | A deterministic CPU compositor is golden-frame testable on any machine (no GPU), gets composited frames (and thus viewable/encodable output) working now, and the spec's GPU requirement (§3.1) is a preview-performance concern that drops in later with no consumer change. |
| 2026-06-22 | **Render-core contracts added to `chroma-core-api` (still draft): `Compositor` + `SourceImage`, and `SceneStyle`/`Shadow`/`Border`; `Project` gains a `scene` field; `Size` gains `Default`.** | The §4.1 render-core surface (Modifier/Camera/Compositor/FrameSink) was only partially defined; `chroma-render` needs the compositor contract and the scene-styling value types (spec BG-03). Contracts are still v0.1 draft, so extending them is not a CONTRACT-CHANGE event. |
| 2026-06-22 | **`chroma-render::render_frame` does the camera solve inline (weighted average) for M2.** Spring smoothing and the §3.4 cross-fade rule are deferred to `chroma-camera` (M5) behind the same call site. | M2 only needs a static/blended camera; a full spring solver is M5 work. The pipeline shape (source → solve → composite → overlay) is fixed now so M5 slots in without touching `render_frame`'s consumers. |
| 2026-06-22 | **M3 export uses the `ffmpeg` CLI as a subprocess, not libav bindings.** `chroma-media-ffmpeg::FfmpegEncoder` pipes rawvideo RGBA to ffmpeg's stdin; MP4 via libx264 (CRF/bitrate), GIF via the one-shot `palettegen`+`paletteuse` two-pass filter. | Spec §1.1 explicitly allows the subprocess path; it needs only the system `ffmpeg` binary (no native linking, no Rust deps — minimal-deps). Verified end-to-end on Linux (H.264 MP4 + GIF). Hardware encode (VAAPI/NVENC, EXP-08) is a later `build_args` addition. |
| 2026-06-22 | **The encoder drains ffmpeg's stderr on a dedicated thread.** | Writing frames to ffmpeg's stdin while its stderr pipe fills would deadlock; a reader thread keeps it drained and lets `finish` report the log tail on failure. |
| 2026-06-22 | **M4 effects each implement `Modifier`; `chroma-modifiers::build` bridges `ModifierSpec` → `Box<dyn Modifier>`.** CropZoom/CursorFollow are camera modifiers; Text/Highlight are overlays. Easing is composed free functions, not inheritance. | Realizes the data→behaviour split (DECISIONS 2026-06-22): the project stores serializable specs, the factory constructs behaviour, the render loop is untouched (open/closed). Crop-zoom + text verified end-to-end to MP4 on Linux. |
| 2026-06-22 | **`chroma-app` is the composition root: a `chroma` CLI** depending on the implementation crates (the only crate allowed to). `render` = synthetic demo → styled video; `record` = X11 desktop → cursor-follow styled video. | Gives a runnable app now without waiting for the Tauri/SvelteKit GUI. `record` works only on native X11 (rootless Xwayland/WSLg can't grab the desktop); the `render` path is verified anywhere ffmpeg exists. |
| 2026-06-22 | **M5 spring camera deferred:** cursor-follow currently targets the raw cursor (no smoothing); `render_frame`'s camera solve stays stateless. | A critically-damped spring (CAM-02) needs per-frame state — a `chroma-camera` solver that threads state through the render loop. Deferred to keep M4 shippable; it composes in behind `render_frame` without changing the effect modifiers. |
| 2026-06-22 | **M5 camera smoothing is a `CameraSmoother` trait in `chroma-core-api`** (draft contract), injected into `render_frame` as `&mut dyn CameraSmoother`. The render core computes the instantaneous blended *target* (renamed `solve_camera_target`); the smoother maps target→applied camera and owns the only per-frame state in the pipeline. `PassthroughSmoother` (the identity, in core-api) is the stateless default. | Keeps spring state out of the otherwise-pure render core and respects the contract rule (smoothing strategies depend on `-api` only): `chroma-camera`'s `SpringSmoother` and a future wgpu-preview smoother both slot in behind the same trait. `render_frame` gains one argument; all consumers are updated (demos/tests pass `PassthroughSmoother`). |
| 2026-06-22 | **`chroma-camera::SpringSmoother` uses the closed-form critically-damped spring** (damping ratio = 1), per scalar component (center x, center y, zoom), sharing one natural frequency; it derives `dt` from `EvalContext::time` and snaps on the first frame. | The analytic solution is exact and unconditionally stable at any `dt` (no overshoot, no blow-up at large steps) — unlike explicit Euler — so framing is deterministic (EXP-06) and identical in preview and export. Frequency is the only knob (CAM-02); `reset()` re-snaps after a timeline seek so the camera never smears across a jump. |
| 2026-06-22 | **M6 keyframes are a `Track` value type in `chroma-core-api`** (`Keyframe { time, value, easing }`, `Easing { Hold, Linear, Smooth }`), sampled with endpoint clamping; the *earlier* key's easing governs each segment. A new `KeyframeCameraModifier` (in `chroma-modifiers`, `ModifierParams::KeyframeCamera { center_x, center_y, scale, weight }`) drives the camera from three tracks. | The project model needs an authored-animation primitive the GUI edits (EDT-06) without coupling to any modifier impl — so it's plain serializable data in the contract crate, sampled at render time, exactly like `ModifierSpec`. Empty tracks fall back to identity-camera values; the spring smoother still glides the sampled result. |
| 2026-06-22 | **M6 persistence/editing lives in a new `chroma-project` crate** (depends on `-api` + serde only): versioned JSON `save`/`load` with a `migrate()` hook (`CURRENT_VERSION = 1`), an undo/redo `History` of reversible `EditCommand`s, and built-in `Preset` looks. | Keeps the data layer the GUI sits on free of render/encode deps (it composes those itself). Undo is lossless because `apply_to` returns each command's exact inverse — applying an inverse yields the forward command, so redo falls out for free; out-of-range edits collapse to `Noop` to keep the history total. Presets apply as two `Set*` commands so a look change is ordinary undoable history. |
| 2026-06-22 | **GUI engine split into a headless `chroma-studio` crate (in the workspace) with the Tauri shell as a thin layer.** `Studio` owns the `Project` + `History`, and exposes edits, presets, preview render, and export; the Tauri commands just lock it and return a serializable snapshot. | Makes the editor's actual logic unit-testable on any machine (no window, no webview) — the studio's undo/preview/export are verified in `cargo test`, including a real-ffmpeg export on Linux. The desktop binary, which can't be built/run in CI/WSL, is reduced to marshalling. |
| 2026-06-22 | **The GUI is Tauri 2 + Vite/Svelte 5 (SPA), not SvelteKit; `app/src-tauri` is its own cargo workspace.** Preview frames cross to the webview as `data:image/bmp;base64,…` (a tiny dep-free BMP encoder), and `app/src/lib/api.ts` falls back to an in-browser mock studio when not running under Tauri. | A Vite+Svelte SPA is the same stack with far less SSR/adapter machinery to get right in one pass, and Tauri serves a static SPA natively. BMP-over-base64 avoids a PNG dependency and renders directly in `<img>`. The mock keeps the whole UI buildable/usable in a plain browser (verified: `vite build` + `svelte-check` clean). Detaching `src-tauri` into its own workspace keeps the heavy webkit/wry deps out of the core `cargo build`/`test`. Naming gotcha: a `$state` rune must not back a variable named `state` (svelte2tsx reads `$state` as a store subscription). |
| 2026-06-22 | **M7 `chroma-capture-wayland` added as the Wayland backend slot; real impl gated behind an opt-in `portal` Cargo feature.** Default build presents the full session shape (`WaylandScreenCapturer`/`EventSource`/`Session`/`open_session`) but reports `Unavailable`; the ScreenCast+PipeWire / RemoteDesktop+libei flow is documented in `src/portal.rs`, and the `CaptureTarget → SourceType` mapping is real and unit-tested. | Wayland forbids direct pixel reads (the M7 security model); the sanctioned path needs the XDG portals + PipeWire, which require a live Wayland session and `libpipewire-0.3-dev` — impossible to build/verify under WSL or CI. Gating the heavy `ashpd`/`pipewire` deps behind a feature keeps the workspace building and testable everywhere (same philosophy as the X11 non-Linux stub), while the contract slot, clock, and target mapping are in place for the live wiring. The app keeps using the X11 backend (works on XWayland) until the feature is real. |
| 2026-06-22 | **M6 hardware encode is VAAPI, selected by the existing `OutputSpec::hardware` bool** in `chroma-media-ffmpeg::build_args` (no new contract field). HW MP4 uses `-vaapi_device /dev/dri/renderD128`, `-vf format=nv12,hwupload`, `h264_vaapi`/`hevc_vaapi`, and `-qp` (VAAPI has no `-crf`). | EXP-08 with the smallest surface: reuse the bool, keep the software path the default. VAAPI is the broadest Linux GPU path (Intel/AMD); NVENC can be added behind the same flag later. Args are unit-tested without a GPU since `build_args` is pure; the actual encode needs a real VAAPI device (not present under WSL). |
| 2026-06-22 | **M5 synthetic cursor marker + click ripple are overlay `Modifier`s** (`CursorMarker`, `ClickRipple`), added as `ModifierParams` variants and built by the factory like every other effect. The marker is a two-pass (outline + fill) triangle of `fill_rect` slices; the ripple is a fading hollow square of four edge rects. | `CompositePass` exposes only rectangles/text (no circle/path), so both shapes are rasterized from rects — keeping the contract minimal while still drawing a crisp resolution-independent cursor (CAM-05) and click feedback (CAM-06). Data/behaviour split preserved: the project stores the spec, `chroma-modifiers` builds the behaviour. |
